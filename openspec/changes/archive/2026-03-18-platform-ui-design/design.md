## 背景

AI助手资源分享平台需要完整的UI设计系统才能开始前端开发。该平台将服务于希望分享和发现MCP配置和Skills技能包的开发者。设计必须在资源受限的环境下工作（2核2G服务器），并支持移动端、平板和桌面端的响应式布局。

当前状态：没有设计文件。需求已文档化但缺少视觉规范。

关键约束：
- 必须使用Pencil设计工具实现设计到代码的自动化
- 目标用户：开发者（技术导向，注重效率）
- 必须支持响应式设计（移动优先）
- 服务器资源有限，需要优化资源文件
- 设计系统必须可维护和可扩展

## 目标 / 非目标

**目标：**
- 创建包含可复用组件的完整设计系统
- 设计所有核心用户流程（浏览、搜索、上传、管理资源）
- 建立视觉识别（现代、简洁、开发者友好）
- 支持从设计文件自动生成代码
- 支持所有屏幕尺寸的响应式布局
- 创建设计版本管理工作流

**非目标：**
- 管理后台设计（未来阶段）
- 高级分析/报表界面
- 移动原生应用设计（目前仅Web）
- 国际化/本地化设计模式
- 暗色模式（可后续添加）

## 决策

### 1. 设计工具：Pencil

**决策：** 使用Pencil设计工具进行所有UI设计工作。

**理由：**
- 允许Claude直接以编程方式操作设计文件
- 支持自动代码生成（设计 → React/Tailwind）
- 消除设计到代码的转换错误
- 需求变更时可快速迭代
- 保持设计和实现的完美同步

**考虑的替代方案：**
- Figma：需要手动导出和代码转换，无编程访问
- Sketch：仅限macOS，无自动化支持
- Adobe XD：自动化有限，正在停止开发

### 2. 设计风格：现代简约 + 开发者导向

**决策：** 采用简洁现代的美学风格，蓝色主色调（#3B82F6），针对开发者用户优化。

**理由：**
- 开发者偏好功能性、无干扰的界面
- 蓝色传达信任和专业性（开发工具常用）
- 简约风格减少认知负担
- 简洁设计在有限服务器资源下性能更好（资源文件更小）

**关键特征：**
- 充足的留白提升可读性
- 清晰的视觉层次
- 一致的间距和对齐
- 微妙的阴影和边框
- 适当使用图标导航

### 3. 组件架构：原子设计系统

**决策：** 使用原子设计原则构建组件库（原子 → 分子 → 组织）。

**理由：**
- 最大化跨页面复用性
- 确保视觉一致性
- 简化维护和更新
- 与React组件模型对齐
- 减少代码重复

**组件层次：**
- 原子：按钮、输入框、图标、徽章、头像
- 分子：搜索栏、卡片头部、统计显示
- 组织：导航栏、资源卡片、评论区
- 模板：带占位内容的页面布局
- 页面：带真实内容的最终实现

### 4. 响应式策略：移动优先

**决策：** 先设计移动端布局，然后扩展到平板和桌面端。

**理由：**
- 强制优先考虑核心内容
- 确保良好的移动体验（用户群体增长中）
- 向上扩展比向下缩减更容易
- 符合现代Web开发最佳实践

**断点：**
- 移动端：< 640px（单列）
- 平板：640px - 1024px（适当使用2列）
- 桌面：> 1024px（3-4列，侧边栏布局）

### 5. 设计令牌系统

**决策：** 在设计令牌文件中集中管理所有设计值（颜色、间距、字体）。

**理由：**
- 设计值的单一真实来源
- 易于全局更新主题
- 防止不一致
- 直接映射到Tailwind配置
- 支持未来主题化（暗色模式、自定义主题）

**令牌类别：**
- 颜色：主色、辅助色、中性色、语义色（成功、错误、警告）
- 间距：4px基础比例（4, 8, 12, 16, 24, 32, 48, 64）
- 字体：字体族、大小、粗细、行高
- 边框：圆角值、宽度
- 阴影：层级
- 动画：持续时间、缓动函数

### 6. 页面结构：一致的布局模式

**决策：** 所有页面使用一致的 头部 + 内容 + 底部 结构。

**理由：**
- 可预测的导航提升可用性
- 降低用户学习曲线
- 简化实现
- 保持视觉连贯性

**布局模式：**
- 固定头部：Logo、导航、搜索、用户菜单
- 主内容：页面特定内容，一致的内边距
- 底部：链接、版权、社交媒体

### 7. 设计文件组织

**决策：** 按页面/功能分离设计文件，共享组件库。

**文件结构：**
```
designs/current/
├── components.pen          # 共享组件库
├── homepage.pen            # 首页设计
├── mcp-list.pen           # MCP列表页
├── mcp-detail.pen         # MCP详情页
├── skills-list.pen        # Skills列表页
├── skills-detail.pen      # Skills详情页
├── user-center.pen        # 用户中心
└── upload.pen             # 上传界面
```

**理由：**
- 更易于管理和版本控制
- 允许不同页面并行工作
- 减少文件大小和加载时间
- 清晰的关注点分离

## 风险 / 权衡

**[风险]** Pencil工具限制可能约束设计复杂度
→ **缓解措施：** 从简单、经过验证的模式开始；尽早验证工具能力；必要时保留手动编码的后备方案

**[风险]** 设计到代码自动化可能产生次优代码
→ **缓解措施：** 审查生成的代码；建立代码质量标准；按需重构；将自动化作为起点而非终点

**[风险]** 移动优先可能导致桌面端功能不足
→ **缓解措施：** 为桌面端规划增强功能；利用更大屏幕空间；确保核心功能在所有设备上可用

**[风险]** 设计系统可能过度工程化
→ **缓解措施：** 从最小可行组件集开始；根据实际需求迭代；避免过早抽象

**[权衡]** 蓝色主题可能缺乏独特性
→ **接受：** 优先考虑可用性和熟悉度而非独特性；可在后续版本中探索品牌差异化

**[权衡]** 单一设计文件可能变得庞大
→ **缓解措施：** 按页面/功能拆分文件；定期审查和重构；保持组件库精简

## 迁移计划

**阶段1：设计系统基础（第1周）**
1. 创建设计令牌文件
2. 构建核心组件库（按钮、输入框、卡片等）
3. 定义布局模板
4. 生成组件截图供审查

**阶段2：核心页面设计（第2周）**
1. 设计首页
2. 设计MCP/Skills列表页
3. 设计详情页
4. 生成页面截图供审查

**阶段3：用户功能页面（第3周）**
1. 设计用户中心
2. 设计上传界面
3. 设计登录/注册页面
4. 完成所有响应式变体

**阶段4：代码生成和集成（第4周）**
1. 从设计生成React组件
2. 配置Tailwind
3. 集成到前端项目
4. 测试响应式行为

**回滚策略：**
- 所有设计文件版本化在Git中
- 保留设计截图用于对比
- 可随时恢复到之前的设计版本
- 代码生成是非破坏性的（可重新生成）

2. **图片优化策略：** 如何处理用户上传的图片和截图？

   - 需要设计：错误页面、空状态插图、内联错误消息

   - 最低要求：键盘导航、ARIA标签、颜色对比度符合WCAG AA标准
